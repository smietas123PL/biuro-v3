Ogólna ocena: Zaawansowany MVP / Pre-Beta
Projekt robi bardzo dobre pierwsze wrażenie — przemyślana architektura wieloagentowa, imponująca liczba testów (39 plików!), kompletny setup Docker z health-checkami i hot-reload. To wyraźnie więcej niż proof-of-concept. Problem leży jednak głębiej: kluczowe implementacje LLM (OpenAI, Anthropic) to stuby zwracające statyczny string, dziesiątki endpointów API robią tylko res.json({ success: true }) bez zapisu do bazy, brak warstwy autoryzacji oznacza że każdy ma dostęp do wszystkiego, a API keys są w bazie w plain-text. Projekt jest świetnym szkieletem, któremu brakuje jeszcze jednego porządnego sprintu implementacyjnego przed uznaniem go za gotowy do produkcji.

Architektura i projekt systemu

7/10
Express 4 · ESM React 18 + Vite Zustand SQLite (WAL) better-sqlite3 @google/generative-ai

✓ Mocny punkt Wzorzec Heartbeat — przetwarzanie agentów w tle
services/heartbeat.js realizuje pętlę co 5 sekund: idle-agent → fetch task → call LLM → check approval rules → detect delegation (@mentions) → detect feedback loop → save response → deduct budget → auto-shutdown. To solidny, przemyślany flow agentowy z poprawną kolejnością kroków.
✓ Mocny punkt Schemat bazy — kompletny i bogaty
14 tabel (companies, agents, goals, tasks, audit_log, tools, agent_tools, responses, templates, company_approval_rules, daily_cost_snapshots, budget_alert_log, prompt_versions, prompt_templates) z foreign keys, UNIQUE constraints i WAL mode. Poziom modelowania godny produkcji.
✓ Mocny punkt Rozszerzalność LLM — multi-provider routing
callLLMWithTools() routuje po prefiksie modelu: gpt-*, claude-*, gemini-*, demo-*. Tryb demo jest w pełni funkcjonalny z polskimi przykładowymi responsami. Jedyny realnie działający provider to Gemini.
✗ Krytyczny OpenAI i Anthropic to stuby — nie wywołują API
Obie funkcje mają ciała, ale zwracają statyczny string zamiast wywołania API. Użytkownik konfigurujący GPT-4 lub Claude nigdy nie dostanie prawdziwej odpowiedzi.
async function callOpenAI(agent, task, tools) { const apiKey = agent.api_key || process.env.OPENAI_API_KEY; if (!apiKey) throw new Error('Missing OpenAI API Key'); return `[OpenAI ${agent.model}] Analyzed and processed: ${task.description}. Used tools: none.`; // ↑ STUB — nigdy nie wywołuje https://api.openai.com }
✗ Krytyczny Dziesiątki endpointów API to "no-ops"
Np. PATCH /agents/:id/budget, PATCH /agents/:id/active, PATCH /agents/:id/parent, DELETE /agents/:id (nie usuwa z bazy), PATCH /tasks/:id/priority, POST /agents/:id/tools, DELETE /agents/:id/tools/:tool_id — wszystkie zwracają { success: true } bez dotknięcia bazy danych.
⚠ Uwaga Template apply hardkoduje companyId = 1
router.post('/templates/:id/apply') zawiera const companyId = 1; // Default. W środowisku multi-company to poważny błąd — każdy szablon zawsze nadpisze dane pierwszej firmy.
⚠ Uwaga Potencjalne nieskończone pętle w heartbeat
Delegacja via @mentions tworzy nowe zadania bez limitu głębokości. Podobnie słowo kluczowe FEEDBACK w odpowiedzi LLM tworzy kolejne zadanie. Brak max_delegation_depth może doprowadzić do eksplozji zadań i nieskończonej pętli kosztów.
ℹ Info Post-factum approval logic jest koncepcyjnie odwrócona
System wywołuje LLM, a następnie sprawdza reguły kosztowe. Jeśli koszt przekracza próg, zadanie jest "blokowane" — ale pieniądze już zostały wydane. Pre-approval check powinien być przed wywołaniem LLM.

Bezpieczeństwo

5/10
✗ Krytyczny Brak jakiejkolwiek autoryzacji/autentykacji
Każdy kto ma dostęp do serwera może czytać i modyfikować dane wszystkich firm, agentów i zadań. Brak JWT, session, API key middleware — nawet podstawowego. Przed deployment na publiczny serwer to must-have.
✗ Krytyczny API Keys agentów w plain-text w SQLite
Kolumna agents.api_key TEXT przechowuje klucze OpenAI/Anthropic/Google bez szyfrowania. Dump bazy = wszystkie klucze ujawnione. Należy używać AES-256-GCM z kluczem z env lub dedykowanego vault (np. HashiCorp Vault / AWS Secrets Manager).
✗ Krytyczny CORS wildcard — brak restrykcji origin
app.use(cors()) bez konfiguracji pozwala na requesty z dowolnej domeny. W produkcji: cors({ origin: process.env.FRONTEND_URL }).
⚠ Uwaga vm.Script nie jest prawdziwym sandboxem
Node.js vm module nie izoluje całkowicie — znane są techniki ucieczki z kontekstu vm. Do bezpiecznego eval wyrażeń matematycznych lepiej użyć mathjs (ma własny bezpieczny parser) lub isolated-vm.
// Obecne (ryzykowne): const script = new vm.Script(`result = ${args.expression};`); // Zalecane: import { evaluate } from 'mathjs'; const result = evaluate(args.expression);
✓ Dobra praktyka Whitelist komend bash + path traversal guard
Tool bash ma whitelist: ['echo', 'date', 'ls', 'cat', 'pwd', 'wc', 'head', 'tail']. file_read / file_write sprawdzają czy ścieżka nie wychodzi poza WORKSPACE_DIR. Dobry fundament.
✓ Dobra praktyka HTTPS-only dla api_call + ograniczenie rozmiaru odpowiedzi
Tool api_call odrzuca URL bez https:// i tnie odpowiedź do 1000 znaków. Chroni przed SSRF i exfiltrацją.
⚠ Uwaga API Keys w store Zustand → localStorage
Test w store.test.js potwierdza: JSON.parse(localStorage.getItem('apiKeys')). Klucze API w localStorage są dostępne dla każdego skryptu na stronie (XSS). Rozważyć sessionStorage lub backend-side storage.
ℹ Info Brak rate limitingu na API
Brak express-rate-limit — endpoint /api/tasks można wywołać 1000x/s i zasypać agentów zadaniami. Rate limiter istnieje w schemacie bazy (tabela agent_rate_limits), ale nie jest podłączony do API warstwy HTTP.
ℹ Info Brak walidacji wejścia (Zod / Joi)
Endpointy jak POST /agents przyjmują req.body bez walidacji. Brak sprawdzenia czy model ma właściwy format, czy api_key nie jest pusty, czy company_id istnieje w bazie. Podatne na SQL injection przez concatenację (choć better-sqlite3 używa prepared statements, co chroni w większości przypadków).

DevOps i Developer Experience

8/10
✓ Mocny punkt Docker z health-checkami, named volumes i HMR
Backend ma właściwy healthcheck z retries i start_period. Frontend jest zależny od healthy backendu (depends_on: condition: service_healthy). SQLite persystuje w named volume. HMR WebSocket na dedykowanym porcie 24678.
✓ Mocny punkt Oddzielne compose dla dev i prod
docker-compose.yml (dev) używa hot-reload volumes i target dev. docker-compose.prod.yml używa target base, restart: unless-stopped, inny port. Właściwe rozdzielenie środowisk.
✓ Mocny punkt SQLite WAL mode + foreign_keys + busy_timeout
db.pragma('journal_mode = WAL') — lepsza współbieżność dla SQLite. foreign_keys = ON — integralność referencyjna. busy_timeout = 5000 — unika "database is locked" przy równoległych zapisach z heartbeat.
⚠ Uwaga Prod frontend serwowany przez vite preview
docker-compose.prod.yml uruchamia npm run build && npm run preview. Vite preview nie jest przeznaczony do produkcji — brak gzipowania, nagłówków cache, security headers. W produkcji: zbudować statyczne pliki i serwować przez Nginx.
⚠ Uwaga Brak HSTS, CSP, X-Frame-Options
Żaden z serwerów (Express ani Vite) nie ustawia security headers. Minimum to helmet() dla Express w produkcji.
ℹ Info SQLite nie skaluje się horyzontalnie
Dla jednego serwera i jednej firmy SQLite jest w porządku. Przy multi-instance deployment (load balancer) i wielu firmach należy rozważyć PostgreSQL + libSQL (Turso) lub PlanetScale.

Jakość kodu i wzorce

7/10
✓ Mocny punkt Spójna architektura ESM w całym projekcie
Zarówno backend jak i frontend używają ES Modules ("type": "module"). Import/export везде — brak mieszania CJS i ESM, co eliminuje klasyczne pułapki Node.js.
✓ Mocny punkt Tryb demo z polskimi przykładami — dobry UX onboarding
getDemoResponse() generuje realistyczne polskie odpowiedzi dla różnych ról agentów (Researcher, ContentWriter, Analyst). Użytkownik może zobaczyć jak system działa bez podawania kluczy API.
⚠ Uwaga Leniwe ładowanie komponentów — brak code-splitting
App.jsx importuje wszystkie 22 komponenty statycznie na górze. Przy rozrastaniu się projektu: React.lazy(() => import('./components/CostDashboard')) z Suspense znacznie zmniejszy initial bundle.
⚠ Uwaga Brak TypeScript
Cały projekt w plain JavaScript. Przy 14 tabelach SQLite i złożonym state (Zustand) TypeScript znacznie ułatwiłby refaktoring i wykrywanie błędów. Migracja stopniowa przez allowJs: true w tsconfig jest możliwa.
⚠ Uwaga Koszt LLM — uproszczone kalkulacje
OpenAI/Anthropic mają cost = 0.01 (hardcoded). Gemini liczy koszt jako 0.001 per call niezależnie od tokenów. Rzeczywiste kalkulacje wymagają zliczania tokenów z odpowiedzi API i mnożenia przez ceny per 1M tokenów.
ℹ Info Brak ESLint / Prettier
Brak pliku .eslintrc ani .prettierrc. W projekcie wieloosobowym formatowanie kodu może się rozjechać. Vitest nie zastępuje lintera.

Plan działań (priorytetyzacja)

P1
Zaimplementuj prawdziwe callOpenAI() i callAnthropic()
Użyj oficjalnych SDK: openai npm package i @anthropic-ai/sdk. Bez tego core product nie działa dla 2 z 3 głównych providerów.
P1
Dodaj middleware autoryzacji (JWT lub API Key)
Minimalnie: express-jwt lub własny middleware sprawdzający Bearer token. Bez auth produkcja jest wykluczona.
P1
Zaszyfruj API keys agentów w bazie
AES-256-GCM z kluczem z ENCRYPTION_KEY env. Encrypt przy zapisie, decrypt przy odczycie w callLLMWithTools().
P2
Uzupełnij stub endpointy API (PATCH, DELETE)
Przynajmniej: budget, active flag, parent_id agenta, priority/archive zadań, agent tools CRUD. Bez tego UI nie może niczego realnie zmienić.
P2
Dodaj limit głębokości delegacji + max retries dla FEEDBACK
Np. max_delegation_depth = 5 i max_feedback_iterations = 3 per parent_task_id. Chroni przed pętlą nieskończoną i runaway costs.
P2
Dodaj testy do CI pipeline
W docker-build.yml dodać: docker compose run --rm backend npm test i docker compose run --rm frontend npm test.
P3
Zastąp Vite preview Nginx w prod
Multi-stage Dockerfile: build step → skopiuj dist/ do obrazu nginx:alpine. Dodaj gzip, cache headers, security headers.
P3
Walidacja wejścia Zod na wszystkich endpointach
npm install zod i schematy dla każdego body. Zwracaj 400 z czytelnym błędem zamiast crashować na undefined.
P3
Napraw pre-approval logic w heartbeat
Sprawdź reguły kosztowe PRZED wywołaniem LLM (np. na podstawie estimated_cost lub wymuszaj approval dla kosztownych modeli). Obecna logika jest post-factum.
P4
Dodaj TypeScript, ESLint, Prettier
Stopniowa migracja: allowJs: true, zacząć od plików serwisów (llm.ts, tools.ts, heartbeat.ts). Dodać eslint-config-airbnb-typescript.
